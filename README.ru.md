## Принцип, лежащий в основе Falcone

Почти каждому продукту нужна одна и та же backend-инфраструктура: база данных,
файловое хранилище, аутентификация пользователей, фоновые задачи, шина событий,
обновления в реальном времени. Строить и обслуживать эту инфраструктуру **один раз
на приложение — и затем заново для каждого клиента —** это именно то место, где
команды теряют время и где рождаются инциденты безопасности.

Falcone существует, чтобы решить это раз и навсегда. Это **мультитенантный BaaS**:
единая платформа, обслуживающая множество изолированных тенантов, у каждого из
которых свои данные, идентичности и ресурсы, доступные через согласованный API.

Две идеи скрепляют всю систему:

1. **Изоляция тенантов — это контракт, а не функция.**
   Каждое чтение и каждая запись ограничены по `tenant_id` (и на уровень ниже — по
   `workspace_id`). Идентичность определяется на границе из токена, передаётся как
   явный контекст через шлюз, сервисы, слой данных и фоновые задачи и применяется в
   базе данных через row-level security и схемы на каждого тенанта. Утечка между
   тенантами считается кардинальной ошибкой.

2. **Возможности выдаются по плану и применяются повсюду.**
   То, что может делать тенант — SQL, реальное время, вебхуки, функции, Kafka,
   хранилище — это пересечение его **коммерческого плана**, **профиля развёртывания**
   и **окружения**. Шлюз ограничивает маршруты по этим ключам возможностей,
   квоты ограничивают потребление по тенанту/рабочему пространству, и каждый отказ
   фиксируется в аудите.

В результате получается платформа, на которой клиент получает полноценный backend за
минуты, а оператор сохраняет единую управляемую и наблюдаемую поверхность — вместо
флота вручную собранных backend-ов.

### Как всё устроено вместе

```
                        ┌──────────────────────────────────────────┐
   Bearer JWT  ──▶  API-шлюз (APISIX)   /v1   идемпотентность, CORS, │
                    разрешает тенант ▸ внедряет идентичность, corr-id │
                        └───────────────┬──────────────────────────┘
                                        ▼
                        ┌──────────────────────────────────────────┐
                        │ control-plane  — 249+ REST-эндпоинтов     │
                        │ tenants · workspaces · auth/IAM · pg ·    │
                        │ mongo · storage · events · functions ·    │
                        │ metrics · plans · quotas · backup         │
                        └───────────────┬──────────────────────────┘
            ┌───────────────────────────┼─────────────────────────────┐
            ▼                           ▼                             ▼
   provisioning-orchestrator   realtime-gateway / webhook-engine   cdc-bridges
   (саги, appliers)            scheduling-engine / backup-status   (pg & mongo → Kafka)
            │                           │                             │
            ▼                           ▼                             ▼
   ┌────────────────────────────────────────────────────────────────────────┐
   │ PostgreSQL (RLS + схема на тенант) · MongoDB · Kafka · S3/MinIO ·        │
   │ Vault (секреты) · Keycloak (IAM: realm на тенант)                        │
   └────────────────────────────────────────────────────────────────────────┘
```

Платформа — это **монорепозиторий pnpm + Turbo** из сервисов на Node.js (ES-модули)
и веб-консоли на React + Vite, разворачиваемый с помощью Helm в Kubernetes и
расположенный за шлюзом APISIX.

## Возможности

| Область | Что это даёт тенанту |
| --- | --- |
| **Жизненный цикл тенанта** | Создание, приостановка, мягкое удаление и полная очистка тенантов через защищённую машину состояний (`draft → provisioning → active → suspended → soft_deleted`), с панелями управления и двойным подтверждением для разрушительных действий. |
| **Сага провижининга** | Асинхронная оркестрация, которая разворачивает (или сворачивает) тенант во всех областях — realm IAM, namespace Kafka, схема Postgres, MongoDB, namespace хранилища, namespace функций — с предварительными проверками и откатом при сбое. |
| **Рабочие пространства (workspaces)** | Границы суб-тенанта со своим slug, окружением, областью IAM и членством. Клонирование рабочих пространств по явным политикам; разрешение наследования общих vs. специализированных ресурсов. |
| **Аутентификация и IAM** | Вход в консоль через делегирование OIDC, регистрация с состоянием ожидания активации, восстановление пароля. Администрирование Keycloak (realm на тенант) для realm-ов, клиентов, ролей, scope-ов и пользователей. Проверка JWT через кэшированный JWKS с откатом на introspection. |
| **Сервисные аккаунты и OAuth2-приложения** | OAuth2-клиенты и сервисные аккаунты с API-ключами на рабочее пространство, с проверкой HTTPS redirect-URI и лимитами по плану. |
| **PostgreSQL** | Ограниченный по тенанту data API плюс администрирование/управление, захват изменений (CDC), метрики и аудит. Изоляция через row-level security (`app.tenant_id` / `app.workspace_id`) и схемы на тенант. |
| **MongoDB** | Document API на тенант/рабочее пространство, администрирование, change streams, метрики и аудит. |
| **Объектное хранилище** | S3-совместимые бакеты, multipart-загрузки, предподписанные URL, политики доступа, уведомления о событиях и квоты ёмкости на тенант. |
| **События (Kafka)** | Управление топиками и ограниченные по тенанту CDC change streams (`<префикс>.<тенант>.<пространство>`), плюс системные топики аудита/квот/жизненного цикла. |
| **Реальное время** | WebSocket-подписки (`/v1/websockets`) с аутентификацией Bearer-JWT, применением scope к каналу и изоляцией тенанта на каждую сессию. |
| **Функции** | Serverless-функции с версиями, активациями, вызовами, откатом и триггерами cron / Kafka / storage. |
| **Вебхуки** | Подписанная доставка вебхуков с повторами и защитой от SSRF (приватные, loopback, link-local и ULA диапазоны блокируются и перепроверяются в момент доставки). |
| **Планирование** | Cron-задачи с квотами на параллелизм и число задач на рабочее пространство и полным аудитом выполнения. |
| **Планы и квоты** | Коммерческие планы отображаются в ключи возможностей, значения квот по умолчанию и профиль развёртывания. Квоты применяют режимы hard-block / soft-grace / soft-exhausted на тенант и рабочее пространство. |
| **Резервное копирование и восстановление** | Перечисление снапшотов, оркестрация восстановления и симуляция восстановления на момент времени (PITR) поверх адаптеров S3 / Postgres / Mongo. |
| **Наблюдаемость и аудит** | Конвейер аудита на тенант (актор, конверт области, ресурс, действие, результат), передаваемый в Kafka и сохраняемый, с семействами метрик, health-проверками, дашбордами и пороговыми оповещениями. |
| **API-шлюз** | Единая публичная поверхность на `/v1` с обязательными ключами идемпотентности, correlation ID, валидацией запросов и таймаутами/повторами на маршрут. |
| **Веб-консоль** | Админ-интерфейс на React + Vite для тенантов, рабочих пространств, участников, баз данных, хранилища, функций, событий, планов, квот и наблюдаемости. |

## Быстрый старт с Docker Compose

Репозиторий поставляет Compose-стек, который поднимает **реальные backing-сервисы**,
с которыми работает Falcone — PostgreSQL, Keycloak, Redpanda (Kafka), MongoDB
(однонодовый replica set), MinIO (S3) и Vault — плюс шлюз APISIX и action runner.
Это самый быстрый способ получить рабочее окружение на вашей машине.

### Требования

- Docker с плагином Compose (`docker compose`)
- Node.js 20+ и `pnpm` (через `corepack enable`) — нужны только для запуска наборов тестов

### 1. Клонировать и установить

```bash
git clone https://github.com/gntik-ai/falcone.git
cd falcone
corepack enable
pnpm install
```

### 2. Поднять стек с Docker Compose

Вспомогательный скрипт настраивает за вас health-проверки, миграции, replica set
Mongo, бакет MinIO и устройство аудита Vault:

```bash
cd tests/env
./up.sh
```

…или управляйте Compose напрямую, если нужны только контейнеры:

```bash
docker compose -f tests/env/docker-compose.yml up -d --build
docker compose -f tests/env/docker-compose.yml ps
```

### 3. Сервисы и порты

| Сервис | URL / эндпоинт | Учётные данные |
| --- | --- | --- |
| API-шлюз (APISIX) | http://localhost:9080 | Bearer JWT из Keycloak |
| Keycloak (IdP) | http://localhost:8081 | `admin` / `admin` |
| PostgreSQL | `localhost:55432` | `falcone` / `falcone` |
| MongoDB (rs0) | `localhost:57017` | — |
| Redpanda (Kafka) | `localhost:19092` | — |
| MinIO (S3 API) | http://localhost:59000 | `minioadmin` / `minioadmin` |
| Консоль MinIO | http://localhost:59001 | `minioadmin` / `minioadmin` |
| Vault (dev) | http://localhost:58200 | токен `root` |

### 4. Проверить в деле

```bash
# Запустить наборы unit / contract / e2e против живого стека
pnpm test

# или контрактный black-box набор (только публичный интерфейс)
bash tests/blackbox/run.sh
```

### 5. Остановить и удалить

```bash
cd tests/env
./down.sh
# или: docker compose -f tests/env/docker-compose.yml down -v
```

> Для полноценного продакшн-развёртывания (runtime функций, control-plane и
> веб-консоль) используйте Helm-чарты в `helm/` и `charts/` на кластере
> Kubernetes — см. манифесты в `deploy/`.

## Структура репозитория

```
apps/            control-plane (поверхность REST API) · web-console (UI на React)
services/        gateway-config, realtime-gateway, webhook-engine, cdc-bridges,
                 scheduling-engine, provisioning-orchestrator, backup-status,
                 audit, adapters, internal-contracts, …
charts/ helm/    Развёртывание Kubernetes / Helm
deploy/          Маршруты APISIX, bootstrap для kind/OpenShift
tests/           blackbox (контракт) · e2e (Playwright) · env (стек Compose)
openspec/        Рабочий процесс изменений, управляемый спецификациями
```
