# Current-State & Impact Map — Replace HashiCorp Vault with OpenBao

**Stage:** ANALYZE (read-only survey). No design decisions here — this hands the architect a precise map of reality with `path::symbol` evidence.

**Goal of the change:** make every FUTURE install of Falcone provision and use **OpenBao** instead of HashiCorp Vault. Falcone itself must change so the default install path no longer pulls `hashicorp/vault`.

> Code-only rule (CLAUDE.md): all behavioral facts below are from source code / chart templates / build files. Repository docs (`README*`, `docs/`, `docs-site/`) are listed ONLY as "doc-only — needs later update"; they are NOT used as a source of truth and any doc-only claim is marked `⚠ not code-verifiable`.

## Summary

- **Install source of truth = a hand-written (vendored) Helm subchart** at `charts/in-falcone/charts/vault/`, declared as the umbrella dependency `vault` (version `0.1.0`, `repository: file://./charts/vault`) in `charts/in-falcone/Chart.yaml`. It is **not** a remote HashiCorp chart — it is 17 local template/values files plus a packaged `charts/in-falcone/charts/vault-0.1.0.tgz` and a `Chart.lock` entry. The image is pinned to `hashicorp/vault:1.15.0` in `charts/in-falcone/charts/vault/values.yaml`.
- **Vault is DISABLED by default** (`vault.enabled: false`, `eso.enabled: false` in `charts/in-falcone/values.yaml`). It is opt-in via the kind overlay `deploy/kind/values-kind-vault.yaml` (self-signed TLS path). The base/default render is Vault-free — proven by `tests/blackbox/deploy-completeness-vault-prometheus.test.mjs`.
- **No Vault SDK is a declared dependency anywhere.** Falcone talks to Vault three ways: (1) **raw HTTP fetch** against the KV v2 REST API in `deploy/kind/control-plane/vault-secrets.mjs`; (2) the **`vault` CLI** inside chart Jobs and ops scripts (`vault-init-job.yaml`, `scripts/verify-secret-storage.sh`, `tests/env/up.sh`); (3) **ESO's built-in Vault provider** (`external-secrets.io/v1beta1` `ClusterSecretStore`) in `charts/in-falcone/charts/eso/`. One test fixture (`tests/hardening/lib/fixtures.mjs`) does a dynamic `import('node-vault')`, but `node-vault` is in NO `package.json` — it only resolves if separately installed.
- **Data in Vault:** a KV v2 mount at `secret/` holding (a) platform/app credentials at `secret/{platform,gateway,iam}/*` consumed by ESO, and (b) per-tenant/per-workspace function secrets at `secret/data/falcone/workspace-secrets/{tenantId}/{workspaceId}/{name}`. No transit / encryption-as-a-service is used in code (encryption master-key is a stored KV value, not transit).
- **Dependents:** ESO ExternalSecrets feeding 6 platform components (apisix/keycloak/documentdb/kafka/postgresql/s3), the kind control-plane workspace-secrets API + function-deploy env injection, the secret-audit-handler sidecar (tails Vault's file audit log), and the secret-rotation lifecycle in `services/provisioning-orchestrator` (which uses an **injected** `vaultClient`, default = no-op stub — never a real client).
- **Verdict: largely drop-in.** OpenBao is a Vault fork, API/CLI-compatible at these versions (KV v2 REST, `vault operator init/unseal`, k8s auth, audit file device, ESO Vault provider all map 1:1). The change is overwhelmingly **swap the image + CLI binary + naming**, not a client rewrite. The real work is the install source of truth (the vendored subchart: image repo, names, TLS SANs, audit log path, init script), the ESO provider stanza (still `vault:` provider type — verify OpenBao is accepted by ESO 0.9.0 at the chosen version), and a long tail of identifiers/labels/paths that say "vault". Nothing found is Vault-Enterprise-only.

## Deployment Objects & Config

When `vault.enabled=true` (+ `eso.enabled=true`), the chart renders, in namespace **`secret-store`** (ESO objects in `eso-system` and the 6 app namespaces):

**Vault subchart objects**
- `StatefulSet/vault` (1 replica) — image `hashicorp/vault:1.15.0`, `server -config=/vault/config/vault.hcl`, TLS listener on :8200, cluster :8201, audit sidecar container.
- `ConfigMap/vault-config` — `vault.hcl`: **file storage** (`/vault/data`), TLS listener, `ui=true`, lease TTLs, api/cluster addrs.
- `Service/vault` (ClusterIP, 8200/8201) + `Service/vault-internal` (headless).
- `PersistentVolumeClaim/vault-data` (10Gi) + `PersistentVolumeClaim/vault-audit` (2Gi).
- `ServiceAccount/vault`, `ServiceAccount/eso-vault-auth` (in eso-system), `ClusterRole/ClusterRoleBinding vault-kubernetes-auth`.
- `NetworkPolicy/vault-access-policy`.
- `Job/vault-init` (bootstrap) + `Job/vault-migration` (idempotent stub).
- 5× `ConfigMap/vault-policy-{platform,tenant,functions,gateway,iam}` (HCL policies).
- TLS: either `Certificate/vault-server-tls` (cert-manager mode) **or** the pre-install hook `Job/vault-tls-bootstrap` (+ Role/SA/RoleBinding) writing `Secret/vault-server-tls` (self-signed; the kind path).

**ESO subchart objects**
- `external-secrets` deployment (from the remote `external-secrets` 0.9.0 dependency).
- `ClusterSecretStore/vault-backend` — `provider.vault` (k8s auth, `path: secret`, `version: v2`, caProvider → `vault-server-tls`).
- 6× `ExternalSecret` (postgresql/documentdb/kafka/s3/apisix/keycloak) → 6 k8s Secrets `*-credentials`.
- `NetworkPolicy/eso-to-vault`.

**Listeners / storage / seal / TLS**
- Listener: single TCP :8200, TLS **on** (`tls_disable = 0`), cert/key at `/vault/tls/`.
- Storage backend: **file** (`storage "file"`), not Raft/Consul. (OpenBao supports the `file` backend.)
- Seal: **Shamir** (`unsealMethod: shamir`, 5 shares / 3 threshold) — manual/automated unseal via the init Job; **no auto-unseal/KMS** (no Enterprise/cloud seal in config). Unseal keys + root token are generated in the init Job and **not persisted** to a Secret (printed/used in-Job only). This is an operational gap to note for migration.
- TLS: cert-manager `Certificate` (default) or openssl self-signed hook (kind). DNS SANs are the Vault Service names.

**Bootstrap / init performed by `vault-init`**
1. `vault operator init` (5/3), parse root token + 3 unseal keys, `vault operator unseal` ×3.
2. `vault auth enable kubernetes` + `auth/kubernetes/config` (k8s host + CA + tokenreviewer JWT).
3. `vault secrets enable -path=secret kv-v2`.
4. `vault audit enable file file_path=/vault/audit/vault-audit.log`.
5. Write 5 policies; create k8s auth roles `platform-role`/`gateway-role`/`iam-role`/`eso-role` (bound SA namespaces).
6. Seed dummy KV values at the 7 platform paths (placeholders, intended to be overwritten by the migration job / real ops).

## Data Held (secret paths / structure) — drives the migration

**KV v2 mount: `secret/`** (engine enabled at `-path=secret`). KV v2 REST exposes data under `secret/data/<path>` and metadata under `secret/metadata/<path>`.

**(A) Platform / app credentials** — written by `vault-init` (dummy seeds), read by ESO:
| Logical path (KV) | Keys | Consumer (ExternalSecret → k8s Secret → ns) |
|---|---|---|
| `platform/postgresql` | `root-password`, `app-password` | `platform-postgresql-credentials` → `postgresql` |
| `platform/documentdb` | `root-password`, `app-password` | `platform-documentdb-credentials` → `documentdb` |
| `platform/kafka` | `admin-password`, `inter-broker-secret` | `platform-kafka-credentials` → `kafka` |
| `platform/s3` | `access-key`, `secret-key` | `platform-s3-credentials` → `s3-compat` |
| `platform/encryption` | `master-key` (32-byte) | not via ESO — referenced by `deploy/k8s/encryption-config.yaml` (k8s at-rest aescbc key) |
| `gateway/apisix` | `admin-key`, `dashboard-password`, `etcd-password` | `gateway-apisix-credentials` → `apisix` |
| `iam/keycloak` | `admin-password`, `db-password` | `iam-keycloak-credentials` → `keycloak` |

**(B) Per-tenant/per-workspace function secrets** — written/read by the kind control-plane (`vault-secrets.mjs`):
- Path: `secret/data/falcone/workspace-secrets/{tenantId}/{workspaceId}/{name}`, value shape `{ value: <string> }`.
- Isolation is **path + credential-derived** (tenantId/workspaceId from the verified principal, never from input). KV v2 versioning is used (`getMeta` returns version; re-set increments).

**(C) Rotation-domain secrets** — the rotation lifecycle (`services/provisioning-orchestrator`) reasons over paths `tenant/<tenantId>/<name>` and `platform|gateway|iam|functions/<name>` and stores the **Vault KV version number** in Postgres (`secret_version_states.vault_version`, `secret_propagation_events.vault_version`). The actual Vault write is via the injected `vaultClient` (stub by default).

**(D) tests/hardening fixture** writes `tenant/{tenantId}/hardening-{runId}` (active/rotated/webhook) via `node-vault` — test data only.

**No transit / encryption-as-a-service.** The encryption master key is a *stored KV value*, not a Vault transit key. No `transit/` mount, no `vault.encrypt`/`decrypt` calls anywhere (grep-confirmed). Migration is therefore a **KV-v2 data copy** (platform paths + any live workspace-secrets), not a transit-key re-wrap.

## Compatibility Assessment (drop-in vs must-change)

OpenBao is a fork of HashiCorp Vault and is API/CLI-compatible at these versions. Mapping what Falcone actually uses:

**Drop-in (no behavioral code change; backend speaks the same protocol)**
- KV v2 REST surface (`/v1/{mount}/data|metadata/...`, `?list=true`) — `vault-secrets.mjs`. OpenBao serves the identical paths.
- `X-Vault-Token` / `X-Vault-Namespace` request headers — accepted by OpenBao.
- `vault operator init/unseal`, `auth enable kubernetes`, `secrets enable kv-v2`, `audit enable file`, `policy write`, `kv put`, `kv list/get`, `token lookup` — all present in OpenBao's CLI (`bao`, and the Vault-compatible CLI). The init Job's CLI calls map 1:1.
- HCL server config (`storage "file"`, `listener "tcp"`, lease TTLs, `api_addr`/`cluster_addr`) — OpenBao uses the same config format and supports the `file` storage backend.
- ESO `provider.vault` (`version: v2`, k8s auth) — ESO's Vault provider targets the Vault HTTP API that OpenBao implements. **Verify** the ESO chart/version in use (`external-secrets` 0.9.0) accepts an OpenBao server with the `vault:` provider type (expected yes; flag for live check).
- Vault file **audit-log JSON** parsed by `vault-log-reader.mjs` — OpenBao inherits the same audit schema (`request.path/operation/id`, `auth.metadata.service_account_*`, `error`, `time`). **Verify byte-compatibility** (highest schema-coupling).
- `node-vault` (test fixture only) — talks the Vault HTTP API; expected drop-in if `node-vault` is ever installed.
- DB `vault_version` columns + `vaultVersion` API/UI fields — store a KV version integer; backend-agnostic, function unchanged.

**Must-change (install source of truth + naming)**
- **Image:** `charts/in-falcone/charts/vault/values.yaml` `vault.image.repository: hashicorp/vault` / `tag: 1.15.0` → OpenBao image (e.g. `openbao/openbao` / `quay.io/openbao/openbao`, version TBD by architect). Also `Chart.yaml appVersion`.
- **CLI binary in Jobs/scripts:** `vault-init-job.yaml`, `vault-migration-job.yaml`, `scripts/verify-secret-storage.sh`, `tests/env/up.sh` invoke the `vault` binary. The OpenBao image's CLI is `bao` (it also historically provides a `vault`-named compatibility binary — **verify which the chosen image ships**); commands/flags are identical otherwise.
- **tests/env dev image:** `tests/env/docker-compose.yml` `hashicorp/vault:1.18 -dev` (env `VAULT_DEV_ROOT_TOKEN_ID`, `VAULT_DEV_LISTEN_ADDRESS`) → OpenBao dev server (OpenBao uses `BAO_DEV_*` equivalents AND retains `VAULT_*` compatibility — verify).
- **Packaged artifacts to regenerate:** `charts/in-falcone/charts/vault-0.1.0.tgz` + `Chart.lock` digest (and `eso-0.1.0.tgz` if the ESO stanza changes) via `helm dependency build`.
- **Tests that assert rendered Vault objects / image:** `tests/blackbox/vault-secrets-backend-kind.test.mjs` and `deploy-completeness-vault-prometheus.test.mjs` will need updating if subchart names/labels/image change.

**Naming / identifiers (design decision — functionally optional for the swap)**
- Subchart dir/name `vault`, Service/StatefulSet/SA/ConfigMap/Job/Policy names (`vault`, `vault-internal`, `vault-config`, `vault-init`, `vault-tls-bootstrap`, `vault-policy-*`, `vault-access-policy`), namespace `secret-store`, TLS Secret `vault-server-tls`, ESO store `vault-backend`, env vars `VAULT_ADDR/TOKEN/KV_MOUNT/NAMESPACE/CACERT/SKIP_VERIFY`, audit path `/vault/data`,`/vault/audit/vault-audit.log`, ESO SA `eso-vault-auth`/role `eso-role`, API enum `secretBackend: "vault"`, fields `vaultMount`/`vaultRequestId`/`vaultVersion`/`vault_version`. Renaming any of these is cosmetic and may break the API/UI contract — architect to decide scope.

**Enterprise / non-OpenBao-compatible flags found:** NONE. No auto-unseal/cloud-KMS seal, no Vault namespaces in use (the `VAULT_NAMESPACE`/`x-vault-namespace` support is dormant), no transit, no Enterprise replication, no Consul/Raft. The integration uses only OSS features OpenBao provides.

## Open Questions / Risks (for the architect)

1. **OpenBao image + CLI binary name** — pick the image (e.g. `openbao/openbao` / `quay.io/openbao/openbao`) and version; confirm whether its CLI is `bao` only or also provides a `vault`-compatible binary. **All chart Jobs + ops scripts shell out to `vault`** — if only `bao` exists, every `vault …` invocation in `vault-init-job.yaml`, `vault-migration-job.yaml`, `scripts/verify-secret-storage.sh`, `tests/env/up.sh` must be rewritten. Also confirm the dev-mode env vars (`VAULT_DEV_*` vs `BAO_DEV_*`) for `tests/env/docker-compose.yml`.
2. **ESO Vault provider vs OpenBao** — does `external-secrets` 0.9.0's `provider.vault` work unchanged against OpenBao, or is an ESO upgrade / a different provider stanza needed? (Expected unchanged; needs a live render/sync check on the kind path.)
3. **Audit-log JSON schema** — confirm OpenBao's file-audit JSON is byte-compatible with what `vault-log-reader.mjs::parseVaultEntry` expects (`request.path/operation/id`, `auth.metadata.service_account_*`, `error`, `time`). The unit test `vault-log-reader.test.mjs` is the regression anchor.
4. **ACL policy templating** — `tenant-policy.hcl.yaml` uses `identity.entity.aliases.auth_kubernetes_cluster_1.metadata.tenantId`. Verify OpenBao's identity/ACL templating + the k8s auth alias metadata key name are identical (the `auth_kubernetes_cluster_1` mount-accessor-derived name could differ).
5. **Naming/contract scope** — decide whether to rename the subchart, k8s object names, env vars (`VAULT_*`), namespace `secret-store`, ESO store `vault-backend`, API enum `secretBackend:"vault"`, and DB/UI `vault_version` fields. Renaming the API enum + DB columns is breaking and orthogonal to the functional swap; recommend keeping them for the first cut.
6. **Migration of existing deployments** — for an already-running Vault: a KV-v2 data copy of `secret/{platform,gateway,iam}/*` + any live `secret/data/falcone/workspace-secrets/**`, and re-seal/re-init under OpenBao. The init Job does NOT persist unseal keys/root token to a Secret (in-Job only), so an existing install's keys are operator-held — the migration runbook must account for that. The `vault-migration-job.yaml` is a stub (no real logic) — it is NOT a usable migration tool today.
7. **Packaged tgz + Chart.lock** — must `helm dependency build` after subchart edits or the render is stale (the packaged `vault-0.1.0.tgz` is what actually ships). Two render-asserting blackbox tests must be updated in lockstep.
8. **Default-off means low blast radius for the swap itself** — because `vault.enabled=false` by default, changing the subchart does not affect the proven default kind render; but the opt-in path (`values-kind-vault.yaml`) + its two render tests + the workspace-secrets contract test (`vault-workspace-secrets.test.mjs`) are the gates that must stay green.

## Impact Summary (sizing)

This is a **medium, install-centric change with a small code footprint and a large naming/test tail.** The install source of truth is a single vendored Helm subchart (`charts/in-falcone/charts/vault/`, ~17 files) plus the ESO consumer subchart and one opt-in overlay (`deploy/kind/values-kind-vault.yaml`); there is no remote Vault chart and no `helm install hashicorp/vault` to repoint. Functionally the swap is close to drop-in: Falcone speaks to Vault only via the KV v2 REST API (one ~167-line raw-fetch client, `deploy/kind/control-plane/vault-secrets.mjs`), the `vault` CLI inside Jobs/scripts, the ESO Vault provider, and a file-audit-log parser — all of which OpenBao implements as a Vault fork, with no Enterprise/auto-unseal/transit/Raft features in use. The genuine work is: (a) the subchart image + CLI binary + dev image + HCL/TLS-SANs/init script, (b) confirming the ESO Vault provider, dev-mode env vars, audit-log JSON, and ACL templating are OpenBao-compatible (live checks), (c) regenerating the packaged `vault-0.1.0.tgz` + `Chart.lock`, and (d) updating the two render-asserting blackbox tests and the workspace-secrets contract test. The data to migrate is a KV-v2 copy (7 platform/app paths + per-tenant/workspace function secrets at `secret/data/falcone/workspace-secrets/{tenant}/{workspace}/{name}`); no transit re-wrap. A long tail of identifiers (`VAULT_*` env, `vault-*` object names, `vaultMount`/`vaultRequestId`/`vaultVersion` contract fields, the `secretBackend:"vault"` API enum, `secret-store`/`vault-backend` names) is cosmetic and can stay to keep the API/UI/DB contracts stable. **File written:** `loop-state/system-changes/replace-vault-with-openbao/impact-map.md`.
