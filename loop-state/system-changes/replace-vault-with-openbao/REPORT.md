## 1. Impact map (full: `impact-map.md`)

Vault was installed exclusively by a **vendored Helm subchart** `charts/in-falcone/charts/vault/` (image `hashicorp/vault:1.15.0`), declared in `charts/in-falcone/Chart.yaml` as a `file://` dependency with `condition: vault.enabled` (**disabled by default**), plus an ESO consumer subchart `charts/in-falcone/charts/eso/`. There is **no remote Vault chart and no `helm install hashicorp/vault`** to repoint. Falcone talks to Vault only via: a raw‑HTTP KV v2 client (`deploy/kind/control-plane/vault-secrets.mjs`), the `vault` CLI inside chart Jobs/scripts, ESO's `provider.vault`, and a file‑audit‑log parser (`services/secret-audit-handler`). The rotation lifecycle uses an injected, default‑no‑op `vaultClient`. Data held: a KV v2 mount `secret/` with 7 platform/app paths + per‑tenant function secrets at `secret/data/falcone/workspace-secrets/{tenant}/{workspace}/{name}`. **No transit / encryption‑as‑a‑service** → migration is a KV‑v2 copy, not a key re‑wrap. **No Enterprise / auto‑unseal / Raft / transit anywhere.** Verdict: largely drop‑in; the work is install‑centric.

## 2. Design (full: `design.md`)

Chosen target: **`openbao/openbao:2.3.1`**, CLI `bao`, canonical env `BAO_*`. Decision: a **full install‑layer rename** `vault` → `openbao` (subchart dir, toggle, k8s object names, ESO store, consumer Secrets, TLS SANs), while **keeping** generic infra (`secret-store` ns, the `X-Vault-Token` header OpenBao honors) and **keeping out of scope** the breaking persisted/public contracts (API enum `secretBackend:"vault"`, DB `vault_version` columns, `vault*` contract fields). Code changes are **additive** (`BAO_* ?? VAULT_*`). Migration/rollout/backup/rollback designed for existing deployments; spec scenarios written against the non‑negotiables, not specific object names.

## 3. What changed in the project (PR #720)

- **Subchart** `charts/in-falcone/charts/vault/` → `charts/in-falcone/charts/openbao/` (image, CLI, `BAO_*` env, `/openbao/*` paths, TLS SANs); umbrella `Chart.yaml` dep + `values.yaml` toggle `openbao.enabled` (**default off**); ESO store → `openbao-backend` (keeps `provider.vault`, `version: v2`); opt‑in overlay `deploy/kind/values-kind-vault.yaml` provisions OpenBao.
- **Code (additive):** `vault-secrets.mjs` reads `BAO_* ?? VAULT_*` (keeps filename + `x-vault-token`); `secret-audit-handler/src/index.mjs` reads `BAO_AUDIT_LOG_PATH ?? VAULT_AUDIT_LOG_PATH`. Out‑of‑scope contract/DB/UI identifiers untouched.
- **Dev env / tests / docs** updated; the packaged `.tgz` + `Chart.lock` are gitignored and regenerated by CI/install.
- **Bootstrap fixes** (pre‑existing — identical on `main`'s Vault subchart — fixed so OpenBao self‑bootstraps): `bitnamilegacy/kubectl:1.32.2`; liveness `sealedcode/uninitcode`; audit sidecar gated behind `openbao.auditSidecar.enabled` (default off; audit file still written); drop invalid ESO `tokenExpirationSeconds`; vendor ESO 0.9.x CRDs + `post-install` hooks + `webhook-wait`; NetworkPolicy DNS + kube‑API (443+6443) egress; robust init‑Job unseal parsing.
- **Migration tooling** under this state dir: `backup.sh`, `migrate.sh`, `rollback.sh`, `RUNBOOK.md`.
- **OpenSpec change** `replace-vault-with-openbao` (validated `--strict`; `openspec/` is gitignored per repo convention so not in the PR file list, but present in the working tree).

## 6. Rollout to an existing running cluster — NOT APPLICABLE

The running `falcone` release on the test cluster has `vault.enabled: false` and **no `secret-store`/`eso-system` namespaces** → no Vault is deployed and no KV data exists, so there is nothing to migrate or cut over. Phase 7 (backup → dry‑run → migrate → verify → rollback) was therefore **not performed**. If a future deployment enables the secret store under Vault and later switches, follow `RUNBOOK.md`: confirm the TEST context, `backup.sh`, `helm diff` dry‑run, **ask the human to confirm**, `migrate.sh`, verify, and `rollback.sh` on failure. Production rollout is done only by merging the PR and re‑running the gated rollout against production with explicit human confirmation.
