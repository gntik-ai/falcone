## Context

The console already has a snippet system (`src/lib/snippets/*` + `<ConnectionSnippets>`) keyed by `ResourceType` with flat token substitution, and a page pattern (`useParams` → `useEffect` fetch via `requestConsoleSessionJson` → render). MCP connect snippets are a different shape (a remote-server config, with a base64-encoded Cursor deeplink), so they are generated by a dedicated function returning the **same `SnippetEntry[]` type** — reusing the renderer without bending the flat-token catalog. The UI is Spanish (no i18n layer), matching the surrounding code.

## Goals / Non-Goals

**Goals:** pure, deterministic connect-snippet generation + a server-detail view-model + an authenticated playground request builder; thin React shell (panel/playground/page + route) over those pure cores; tests that pass on the broken web-console baseline.

**Non-Goals:** the control-plane endpoints themselves (the console calls `/v1/mcp/servers/...`); observability (#398); curation editing (#393); MCP Inspector embedding (a pure-helper playground is enough for the acceptance criteria).

## Decisions

- **Dedicated snippet generator, shared renderer.** `generateMcpConnectSnippets({name, slug, endpoint})` returns `SnippetEntry[]` (Cursor deeplink, Claude Code, claude.ai, VS Code). `<ConnectionSnippets entries={…}/>` already accepts a precomputed `entries` prop, so no change to the existing catalog/generator (their tests stay green). Cursor needs a base64 config in the deeplink — impossible as a flat catalog token — which is the concrete reason for a dedicated generator.
- **OAuth, no embedded secret.** Remote MCP uses Streamable HTTP + OAuth 2.1 (#390); the configs carry only the URL, and every snippet notes that the client completes OAuth. `hasPlaceholderSecrets:false`.
- **Pure view-model + request builder.** `toMcpServerDetailViewModel` tolerates the registry's active-version envelope (#396) or a flat server and yields endpoint/version/source/curated-tools. `buildPlaygroundToolCall` builds the JSON-RPC `tools/call` with the `Authorization: Bearer` from the session and `MCP-Protocol-Version: 2025-11-25` (stable), throwing without a token/endpoint — proving "via OAuth" at the request layer; the network call is the injectable `invoke` in the playground (real default = `invokeMcpTool` through the console session).
- **Eager route import.** The router comment warns lazy console routes throw React #426 in this build; the detail page is imported eagerly like the other wired pages.
- **Verification under the broken baseline.** The web-console suite has a documented pre-existing failing set (global-stub teardown) and CI does not run it; this change is verified by *new tests pass* + *failing set unchanged*, all changes additive (new files + one additive route).

## Risks / Trade-offs

- *Tasks-extension / long-running tools* (a flow tool, #395) in the playground → start with synchronous `tools/call`; long-running Task polling is a follow-up.
- *Snippet drift vs. evolving client config formats* (Cursor/VS Code) → isolated in one pure module, unit-tested, easy to re-pin.

## Migration Plan

Additive: new `src/lib/mcp/`, `src/components/console/mcp/`, one page, one route. No data migration; nothing else changes.
