## Requirements
### Requirement: Console policy builder for user tables

The system SHALL provide an authoring surface in the console that allows a workspace
admin to create, update, and delete RLS policies on user-created (app) tables via a
form accepting `policyName`, `policyMode` (permissive/restrictive), SQL command,
target roles, `usingExpression`, and `withCheckExpression`, with a preview of the
generated SQL before application.

#### Scenario: Policy authored in the console is enforced for an anon caller

- **WHEN** a workspace admin creates a policy on a user table via the console policy
  builder, supplying a `usingExpression` that filters rows by `tenant_id`, and applies
  the policy through the SQL preview + execute flow
- **THEN** a subsequent data-API request authenticated with the anon key returns only
  rows that satisfy the `usingExpression` predicate, and rows belonging to other
  tenants are not visible to the caller

#### Scenario: Policy builder preview shows generated SQL before execution

- **WHEN** a workspace admin fills in the policy form fields
  (`policyName`, `policyMode`, `usingExpression`) and requests a preview
- **THEN** the console renders the `CREATE POLICY` statement generated by
  `buildPostgresGovernanceSqlPlan` before any DDL is executed, and the admin must
  explicitly confirm to proceed

### Requirement: ENABLE ROW LEVEL SECURITY on user tables

The system SHALL allow a workspace admin to enable and force RLS on any user-created
app table in the workspace database, generalising the `table_security` DDL plan path
already used for service tables.

#### Scenario: RLS is enabled on a user table from the console

- **WHEN** a workspace admin activates the `ENABLE RLS` toggle for a user table in
  the console security tab
- **THEN** the system executes `ALTER TABLE ... ENABLE ROW LEVEL SECURITY` on the
  target table, and the security panel subsequently shows `rlsEnabled: true`

#### Scenario: Disabling RLS requires an explicit acknowledgement

- **WHEN** a workspace admin attempts to disable RLS on a user table without
  supplying `disableGuard.acknowledgeTenantIsolationImpact: true` and a
  `disableGuard.reason`
- **THEN** the request is rejected with a 422-class error identifying the missing
  acknowledgement fields and RLS remains enabled

### Requirement: Restrictive versus permissive policy behaviour

The system SHALL evaluate permissive and restrictive policies according to PostgreSQL
RLS semantics: a permissive policy grants access if its predicate passes; a
restrictive policy additionally gates access so that even a passing permissive policy
cannot override a failing restrictive one.

#### Scenario: Permissive policy grants a matching row to the anon role

- **WHEN** a user table has RLS enabled and a single `PERMISSIVE` policy targeting
  the `anon` role with `usingExpression: (tenant_id = current_setting('app.tenant_id'))`
- **THEN** an anon-key request where `app.tenant_id` matches the row's `tenant_id`
  receives the row, and rows with a non-matching `tenant_id` are invisible

#### Scenario: Restrictive policy blocks access even when a permissive policy passes

- **WHEN** a user table has a passing `PERMISSIVE` policy for the `anon` role and
  also a `RESTRICTIVE` policy for the same role whose predicate evaluates to false
- **THEN** the anon-key request is denied for that row (`allowed: false,
  reason: 'rls_filtered'`) because the restrictive policy vetoes the permissive grant

### Requirement: Default-deny when RLS is enabled with no applicable policy

The system SHALL deny all data access to a table when RLS is enabled but no policy
applies to the requesting role, surfacing the `no_applicable_rls_policy` reason in
the console security panel and in API error responses.

#### Scenario: Anon caller is denied when RLS is enabled and no policy exists

- **WHEN** a user table has RLS enabled and no policy targets the `anon` role or
  `PUBLIC`
- **THEN** an anon-key data-API request against that table receives an access-denied
  response and the evaluated reason is `no_applicable_rls_policy`

#### Scenario: Console security panel reflects default-deny state

- **WHEN** a user table has `rlsEnabled: true` and `policyCount: 0`
- **THEN** the console security tab displays a warning indicating that the table is
  in default-deny state and that any data request will be blocked until at least one
  policy is defined

### Requirement: Service role bypasses per its policy predicate

The system SHALL allow a `service_role` policy to be authored with an unrestricted
predicate (e.g. `usingExpression: true`) so that service-key callers retain full
table access while anon-key callers remain row-filtered.

#### Scenario: Service role policy grants unrestricted access

- **WHEN** a user table has RLS enabled, a row-filtering `PERMISSIVE` policy for the
  `anon` role, and a separate `PERMISSIVE` policy for the `service_role` role with
  `usingExpression: true`
- **THEN** a data-API request authenticated with the service key reads all rows
  regardless of `tenant_id`, while an anon-key request on the same table sees only
  the rows permitted by the anon policy

#### Scenario: Service role policy template is available in the policy builder

- **WHEN** a workspace admin opens the policy builder for a user table and selects
  the `service_role` role as the target
- **THEN** the console pre-fills `usingExpression: true` as a default template and
  labels the pattern as the service-role unrestricted access preset
